# Dra01/Dra02 메시지 코드 ↔ 문자 매핑 완전 해석

작성: 2026-07-05 (Claude가 Codex 세션 이어받아 완성)

## 1. 결론

Dra01(창월의 십자가)·Dra02(빼앗긴 각인)의 `message.psb` 텍스트는
**게임 내부 2바이트 코드값**으로 저장되어 있고, 그 코드→문자 매핑의 정답은
**폰트 `LD937714*.DAT` 파일 안에 그대로 내장**되어 있다.
이미지 매칭이나 수작업 대조 없이 100% 정확한 매핑 테이블을 추출했다.

- Dra01: 고유 글리프 코드 1177개 전부 매핑, seed 41자 전수 일치, 누락 0
- Dra02: 고유 글리프 코드 1339개 전부 매핑, 누락 0
- 캐릭터명/아이템명/대사 전 구간 정상 디코딩 확인
  (ソウマ, ヨーコ・ヴェルナンデス / ジョナサン, シャーロット, ＤＳワイヤレス通信 …)

## 2. 메시지 코드 구조

메시지 리소스(`jp_ja` = `#resource#0`)의 각 엔트리는 little-endian uint16 배열.

| 코드 범위 | 의미 |
|---|---|
| `0x8140`–`0x8770`(Dra01) / `0x87xx`(Dra02) | 글리프 코드 (본문 문자) |
| `0x00xx` | 파라미터 (화자 번호, 표정 번호 등; `F0xx` 뒤에 옴) |
| `0xF0xx` | 제어 코드 (`F006`=개행, `F00A`=종결, `F007`=화자, `F003`/`F004`/`F005`=창 제어 추정) |

글리프 코드는 표준 SJIS와 같은 lead/trail 구조를 쓰는 **조밀(dense) 코드**다:

```
trail: 0x40–0xFC, 0x7F 제외  → 행당 188개
dense_rank(code) = (lead-0x81)*188 + (trail-0x40) - (trail>0x7F ? 1 : 0)
```

게임이 실제 쓰는 문자(SJIS 부분집합)를 **SJIS 오름차순으로 나열했을 때의 순번**이
곧 dense_rank다. 즉 코드 공간에 빈 슬롯이 전혀 없다 (Dra01: 0x8140~0x8770 연속 1177개).

## 3. 폰트 DAT 구조와 내장 코드

`LD937714*.DAT` (큰 폰트, 레코드 26바이트 = 2바이트 SJIS 코드 + 12행×2바이트 비트맵):

```
파일 선두부터 26바이트 슬라이싱한 인덱스 n 기준 (Dra01, 1217 레코드):

glyph 0        : SJIS 코드가 파일 끝-4 위치 (0x8DDD)      ← wrap
glyph 1..625   : SJIS 코드가 offset 26n-10 (0x8DDE..0xE95C, 한자 후반부)
glyph 626..665 : 코드 없음 = 특수 그래픽 40개 (메시지 코드 도달 불가)
glyph 666..1216: SJIS 코드가 offset 26n-4  (0x8140..0x8DDD 직전, 기호/가나/한자 전반부)
```

메시지 코드 → 폰트 레코드 인덱스:

```
record = (dense_rank(code) + 666) mod 1217        # Dra01
```

즉 폰트 뱅크는 SJIS 정렬 문자표를 **레코드 666(=0x8140 공백)에서 시작해
파일 끝에서 선두로 감아 도는(rotation)** 배열이고, 도달 불가능한 40개 슬롯에
그래픽 글리프가 끼어 있다. Dra02(1379 레코드, 그래픽 40개)도 동일 구조.

※ 매핑 추출은 rotation 계산조차 불필요: DAT에서 오름차순 SJIS 코드 체인 2개를
스캔해 이어 붙이면 rank 순 문자 리스트가 바로 나온다 (`build_mapping.py`의 `scan_chains`).

## 5. 한글패치에 주는 의미

1. **번역 원문 확보 완료** — 일본어 전문이 나왔으므로 번역 테이블 작업 가능.
2. **역방향 인코딩 규칙 확정** — 한글 텍스트를 넣으려면:
   - 사용할 한글 글자 집합을 정한 뒤, 기존 SJIS 부분집합 슬롯에 배정(완성형/wansung 방식)
   - 메시지 인코딩: `한글자 → 대응 SJIS 슬롯 → dense_rank → 게임코드`
   - 폰트 주입: `record = (dense_rank + 666) mod 1217` 위치의 비트맵 교체
     (레코드의 **내장 SJIS 코드 2바이트는 보존**해야 함 — 기존 exe 패치가 이걸
     코드+비트맵 통째로 덮어써서 글자 위 잡음/매핑 어긋남이 발생했음)
3. **글자 수 제약** — 슬롯 수는 Dra01 1177 / Dra02 1339개. 한글 상용 자모 조합
   필요 글자가 이보다 많으면 스크립트에 실제 등장하는 글자만 추려 배정해야 함
   (VC2 한글패치와 동일한 접근).
4. 작은 폰트 `LD937728*.DAT`(10바이트 레코드 = 2바이트 SJIS 코드 + 8행×1바이트)도
   **같은 내장 코드 구조 확인 완료**:
   - Dra01: 체인 `8140..8353`(176) + `8354..E95C`(1001) = 1177개, 큰 폰트와 동일 문자집합.
     wrap 지점만 다름 (레코드 ~676에서 시작, 총 1257 레코드 중 80개가 코드 없는 슬롯)
   - Dra02: 체인 `8168..E8A6` 1324개 검출 (큰 폰트 1339 대비 15개 차이는
     폰트 주입 단계에서 정밀 확인 필요)
