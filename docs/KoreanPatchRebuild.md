## 기존 exe 패치가 깨졌던 3가지 원인 (모두 해결)

1. **메시지 오프셋 어긋남 (+0x3800)**: 패치 제작 당시의 message.psb는 내부
   리소스 시작이 현재 빌드보다 0x3800 뒤였음. exe는 그 옛 오프셋에 그대로 써서
   일본어 텍스트 중간을 파괴했음. → 현 빌드의 실제 리소스 위치
   (Dra01 res#0 = 0x1E26117C, Dra02 res#0 = 0x208D197C)에 기록.
2. **폰트 회전(rotation) 무시**: 폰트 DAT는 rank r 글리프가 물리 레코드
   (r+666) mod 1217 (Dra01 큰폰트 기준)에 있는 회전 배열인데, exe는 파일
   선두부터 순서대로 덮어씀. → 각 DAT의 내장 SJIS 코드 체인을 스캔해
   rank별 정확한 비트맵 위치에 주입.
3. **내장 코드 파괴**: exe는 [코드2B][비트맵]을 통째로 덮어써 레코드의 코드
   필드를 오염시킴(글자 위 잡음의 원인). → 비트맵 24B(또는 8B)만 교체하고
   코드 필드는 보존. 패치 후에도 코드 체인이 완전히 재스캔됨을 확인.

## 세부 기법

- **패치 리소스 해석**: 기존 exe의 BigFont/SmallFont 레코드 r = 게임 토큰
  dense rank r의 한글 글리프. Dra01 메시지(OrdersData+SentencesData)는 이미
  dense rank 토큰으로 인코딩된 완전한 한국어 번역이었음. Dra02는
  LinkMap/Korean/RomDataMap 픽클로 엔트리별 토큰 재기록 방식.
- **wrap 슬롯 회피**: 비트맵이 파일 경계에 걸리는 레코드(Dra01 big rank 551,
  Dra02 big 551/small 14)는 쓰지 않고, 해당 토큰을 미사용 rank로 리매핑:
  - Dra01: 0x83F0 → 0x86D0 (rank 1083), 토큰 2개
  - Dra02: 0x83F0 → 0x875D (rank 1157), 0x814E → 0x875E (rank 1158), 토큰 17개
- **플랫폼 변형별 처리**: NX/PS5 = 표준 26B/10B 레코드. WIN 큰폰트 =
  [한자 26B 섹션(rank 552+)] + [전체 문자 8×8 10B 섹션] 이중 구조라 각각 주입.
  WIN 작은폰트(코드 체인 없음, 형식 미해독)와 PS5 작은폰트 저영역(기호)은
  원본 유지 — 잘못 건드리는 것보다 안전.
- **기록 안전장치**: 모든 쓰기(12,594건)는 허용 영역(메시지 리소스 2곳,
  폰트 DAT 12곳, 이미지 11곳) 내부인지 검사 후 수행.
